\chapter{Methodology}\label{chapter:methodology}


\section{Problem Statement and Prediction Target}

In this thesis, we address the transfer of the T3 runtime prediction approach from Umbra to PostgreSQL parsed plans.
The central methodological goal is to preserve T3’s good performance~\parencite{t3}.
Thus, we use its main concepts and adapt plan conversion and feature extraction to PostgreSQL-specific plan information accordingly.

The final prediction target is total query runtime.
Each query plan is first decomposed into pipelines and a feature vector is computed for each pipeline.
These per-pipeline feature vectors are then summed into a single query-level feature vector,
from which the model directly predicts the total query runtime in one step.
The pipeline decomposition is therefore used for feature extraction,
not as a separate prediction unit.

\section{Data and Preprocessing}
Regarding the training data, we use the public Zero-Shot benchmark data introduced with the Zero-Shot Cost Model.  
The benchmark is particularly suitable for this thesis because it consists of real-world datasets (e.g., IMDB) and synthetically generated workloads designed to cover diverse query structures.
This includes \emph{SPAJ} (Select-Project-Aggregate-Join) queries with conjunctive predicates.
Overall, the benchmark contains 20 databases with large variation in tables, columns, and foreign-key relationships \parencite{zeroshot}.

Having this multitude of databases allows us to test the generalization capabilities of our model.
Similar to the T3 paper~\parencite{t3}, we can train the model on all but one databases and test on the remaining one.
Moreover, as the widely used JOB workload is included, we can compare our performance predictions to other implementations later on.
Further, PostgreSQL is the intended database system of our implementation, which avoids the effort of building a separate PostgreSQL benchmark.

Following the benchmark construction described in the ZeroShot paper~\parencite{zeroshot}, queries are generated per database and executed on a PostgreSQL DBMS.
Then, they are stored as raw traces (i. e., the output of \texttt{EXPLAIN ANALYZE}).
Next, the authors preprocess the queries to remove very long-running queries (queries above 30\,s are excluded) to keep the benchmark practical and reproducible \parencite{zeroshot}.
A further preprocessing step is to parse the raw traces into structured plan representations.
These \texttt{parsed\_plans} contain the operator nodes found in the raw traces, but organize them in a tree structure with node-wise attributes.
For every node, e.g., operator type, estimated/actual cardinalities, widths, and runtimes are stored in the \texttt{plan\_parameters} field.

The tree structure can be used to create pipelines and the node information will be used to train the model.
Consequently, the parsed plans offer a good starting point for our pipeline and we can begin without further preprocessing.



\section{Plan Representation}

We start from one element, i. e., one plan, of the \texttt{parsed\_plans} array in the benchmark JSON files.
An object in the array is a rooted tree: every node carries \texttt{plan\_parameters} (operator name, estimated and actual cardinalities, 
estimated tuple width, planner costs, optional filter predicates, timings, and further fields) and a list \texttt{children} of child subtrees.


We convert this tree recursively into the same nested operator structure
that the original T3 implementation expects
for Umbra plans~\parencite{t3}.
PostgreSQL operator names are mapped to abstract operators (\texttt{tablescan}, \texttt{join}, \texttt{sort}, and \texttt{groupby} nodes).

Regarding joins, the child order is normalized so that it matches T3's pipeline rules: i. e., the PG-build subtree becomes the \emph{left} child and the PG-probe subtree the \emph{right} child.

Since the feature vector uses the PostgreSQL-native features and operators of the \texttt{parsed\_plans}, we need to keep the original, not converted \texttt{plan\_parameters} available for the feature extraction.
Hence, the converted abstract operator names are only used to match the internal T3 representation for pipeline assignment, but not for the feature extraction.

\section{Pipeline Decomposition}

Umbra plans group operators into \emph{pipelines}: sets of operators that can produce tuples in a pipelined fashion, separated by \emph{pipeline breakers}. 
Namely these are operators/stages, where data is materialized. For instance, sorting, grouping, or materialization, but also the build side of a hash join~\parencite{t3}.

Since PostgreSQL plans do not provide such a pipeline breakdown, we need to derive it from the converted tree.
We assign each node to a pipeline by walking the converted tree with the same high-level rules as in the original T3 implementation~\parencite{t3}.
Thus, PG's sort and aggregate nodes break pipelines; a PG-materialize node is treated as a breaker.
Joins place the build side in one pipeline and keep the join together with the probe side in the other pipeline.

\section{Feature Vector Design}\label{sec:methodology-implementation}

We retain the pipeline decomposition for feature extraction rather than opting for a complete query-level feature vector for three reasons.
By mirroring T3's pipeline-based feature design~\parencite{t3},
we can use the same pipeline-centric features such as operator percentages, which are only well-defined within a pipeline boundary.
Further, following T3's ablation study, which includes per-pipeline predictions, we are able to provide a more thorough comparison
of prediction granularities in Section~\ref{sec:evaluation_ablation}.
Finally, PostgreSQL-specific issues affect the per-pipeline and per-tuple approaches
in our setting, as discussed in the evaluation (Section~\ref{sec:evaluation_ablation});
the decomposition keeps the model extensible so that future work can address those issues.

The original T3 feature mapper for Umbra builds a feature vector from execution phases within each pipeline (e.g.\ scan, build, probe phases) and from cardinalities and tuple sizes attached to those phases~\parencite{t3}.
On the other hand,PostgreSQL plans contain different data and we construct the feature vector for each pipeline after the aforementioned pipeline decomposition.

Once the feature vector is constructed, we can train the model.
Our feature vector differs from the original T3 feature vector in the following ways:



\begin{enumerate}
  \item \textbf{PostgreSQL-native operator features.} The \texttt{parsed\_plans} expose PostgreSQL-native operator nodes. 
  Thus, we add entries for these PostgreSQL-specific operators in the feature vector.
  As no execution stages are available, we aggregate features only per operator.
  Hence, we do not include the stage-based features of the original T3 feature mapper. This avoids inventing synthetic stages for every node.
  \item \textbf{PostgreSQL planner cost estimates.} We include sums and extrema of PostgreSQL's \texttt{est\_cost} and \texttt{est\_startup\_cost} from \texttt{plan\_parameters} in the feature vector.
  These are optimizer-internal estimates of work and startup cost and are an additional PostgreSQL-specific signal that is not available in the Umbra feature vector.
  Consequently, we can include this information in the feature vector.
  \item \textbf{Planned parallelism.} Another PostgreSQL-specific signal is the planned parallelism. So, we take the sum of \texttt{workers\_planned} over all operators in the pipeline.
  Hereby, we include a parallelism signal in our feature vector.
  \item \textbf{Sizes.} T3 uses the size of output tuples in bytes for materializing operators~\parencite{t3}.
  In Umbra, T3 derives the tuple size by summing the per-attribute byte sizes from the operator's IU (information unit) list,
  giving a precise estimate of the tuple byte width.
  PostgreSQL provides a direct semantic equivalent: \texttt{est\_width},
  the planner's estimated average output row width in bytes per node.
  Subsequently, we use \texttt{est\_width} as the tuple size for each operator.
  The main difference to the Umbra approach is that \texttt{est\_width} is a pre-aggregated planner estimate
  rather than a sum of per-attribute type metadata.
\end{enumerate}

In the Umbra feature vector of T3, rows are indexed by both operator type and execution phase (scan, build, probe, \ldots) like \texttt{HashJoin\_Probe\ldots}~\parencite{t3}.
Now with the PostgreSQL implementation, we miss out on the execution phase dimension.

Whereas not an equivalent substitute, we replace this information with a pipeline-level dimension and keep the operator-level features for the individual operators.
The pipeline-level block produces additional metadata in a compact manner, e.g., by using averages and sums over operators.
For instance, we consider planned parallelism, width, and cost estimates per pipeline and not per operator.
Subsequently, the feature vector stays more compact.

Each pipeline's feature vector consists of two concatenated blocks: pipeline-level aggregates and an operator-level block.

In total, the feature vector is fixed and has 153 dimensions (23 pipeline-level features and 130 operator-type features).
Empty features are zeroed to keep the feature vector dimension.

Regarding the cardinalities, either the actual or the estimated cardinalities are used in the feature vector, depending on which is selected in training.

Once the feature vectors for all pipelines are constructed, we get the feature matrix for the whole plan.


\section{Training}

We keep T3's LightGBM decision tree setup~\parencite{t3} and use the newly constructed PostgreSQL-native feature matrix.
A main difference is that we predict the total query runtime directly,
rather than predicting individual pipeline durations.

The training label is the total observed query runtime.
The model input is formed by summing all per-pipeline feature vectors of a query
into a single query-level feature vector.
This per-query prediction approach was found to perform best for our PostgreSQL setting,
as demonstrated in the ablation study (see Section~\ref{sec:evaluation_ablation}).

In T3~\parencite{t3}, the training target is the per-tuple execution time for each individual pipeline instead,
and the predicted pipeline times are summed to obtain the query runtime.
We evaluate all three granularities --- per-query, per-pipeline, and per-tuple ---
in the ablation study and discuss why per-query works best for our case.

Our model uses gradient boosted decision trees as implemented in LightGBM~\parencite{lightgbm}.
The booster is trained with a mean absolute percentage error (MAPE) objective on the training rows,
in line with the original T3 work~\parencite{t3}.

Furthermore, we use the same training parameters as in the original T3 work~\parencite{t3}.
Hence, the pipeline duration targets $t$ are transformed as $-\log(t)$ to stabilize training.
Also following T3, we cap the number of boosting rounds at 200 trees to mitigate overfitting.

Regarding the training data, we mirror T3's train/validation split.
We sample 80\% of the training data for training and 20\% for validation~\parencite{t3}.
The default seed used for shuffling the training data is 42.

Concerning the train/test split, we match the training intention of the Zero-Shot benchmark~\parencite{zeroshot} and of T3's cross-instance evaluation~\parencite{t3}.
Both aim to measure whether the trained model transfers to an unseen database instance.
Thus, we use a leave-one-out split on the \texttt{parsed\_plan} files:
one instance is held out as the test set and all other instances form the training data.


\section{Other Cost Models}

We list models apart from T3 and ZeroShot here, which will be mentioned in the evaluation (Chapter~\ref{chapter:evaluation}).
There, we compare our model to other performance prediction implementations.

\paragraph{Learned Cost Models in the Evaluation.}

\textit{FlatVector}~\parencite{flatvector} is an early machine-learning-based approach
that encodes each query as a flat feature vector of aggregated statistics
and fits a regression model over it.
It makes no use of the tree structure of query plans
and serves primarily as a simple non-neural baseline.

\textit{MSCN}~\parencite{mscn} is a multi-set convolutional network originally proposed for cardinality estimation.
It encodes tables, joins, and predicates as separate sets of feature vectors
and applies a small neural network to each set before combining their outputs.
Also, it can be used as a learned cost model.

\textit{QPPNet}~\parencite{qppnet} is a plan-structured neural network for query performance prediction.
It assigns a separate neural unit to each operator type in the query plan tree,
so that each operator's representation is computed from the representations of its child operators.
Predictions are aggregated bottom-up to produce a single runtime estimate per query.

\textit{E2E}~\parencite{e2e} is an end-to-end learned cost estimator based on tree-structured convolutional networks.
It encodes the full query plan tree by recursively applying a convolutional layer to parent--child pairs of operators
and learns to predict query execution time directly from the encoded plan representation.

\textit{NEO}~\parencite{neo} is a learned query optimizer that uses a value network
to score query plans and guide join ordering.
Its internal cost model is a tree-convolutional neural network
that encodes query plans similarly to E2E, and this cost model is what is evaluated in the comparison.

\textit{QueryFormer}~\parencite{queryformer} is a transformer-based cost model
that encodes query plans as trees using a tree-structured attention mechanism.
By attending over all operator nodes jointly,
it can capture long-range dependencies between operators in a plan,
which is difficult for purely bottom-up tree-convolutional models.

\textit{DACE}~\parencite{dace} is a database-agnostic cost estimator.
It uses an attention-based neural network to encode query plans
and is pre-trained across multiple databases.
Subsequently, it can generalize to unseen database instances
without re-training.

\paragraph{T3-based Models with Different Feature Vectors.}

Wehrstein et al.~\parencite{jobcomplex} also adapted the Umbra T3 model for PostgreSQL.
They use a hybrid feature vector that takes the original Umbra feature vector, and maps matching PostgreSQL operators to Umbra operators.
On top of that, they add additional features that are specific to PostgreSQL, like the Merge Join operator.
However, they do not use fields that account for other relevant PostgreSQL-specific characteristics, like the planned workers or the estimated startup cost fields we use for our model ~\parencite{jobcomplex}.
