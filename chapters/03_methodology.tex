\chapter{Methodology}\label{chapter:methodology}

\section{Problem Statement and Prediction Target}

In this thesis, we address the transfer of the T3 runtime prediction approach from Umbra to PostgreSQL parsed plans.  
The central methodological goal is to preserve T3’s core modeling principles.
Upon that, we adapt plan conversion and feature extraction to PostgreSQL-specific plan information.

Also, the prediction target is total query runtime.  
Following T3, each query plan is decomposed into pipelines and a per-tuple runtime is predicted for each pipeline.
Afterwards, we aggregate the per-tuple and then the pipeline runtimes to obtain the final query-runtime estimate.

\section{Data and Preprocessing}
Regarding the training data, we use the public Zero-Shot benchmark data introduced with the Zero-Shot Cost Model.  
The benchmark is particularly suitable for this thesis because it consists of real-world datasets (e.g., IMDB) and established benchmark workloads (e.g., SSB, TPC-H and JOB).
Overall, the benchmark contains 20 databases with large variation in tables, columns, and foreign-key relationships \parencite{zeroshot}.

Having this multitude of databases allows us to test the generalization capabilities of our model.
Similar to the T3 paper, we can train the model on all but one databases and test on the remaining one.
Moreover, as the widely used JOB workload is included, we can compare our performance predictions to other implementations later on.
Further, PostgreSQL is the intended database system of our implementation, which avoids the effort of building a separate PostgreSQL benchmark.

Following the benchmark construction described in the paper, queries are generated per database and executed on a PostgreSQL DBMS (v12) on c8220 nodes on the CloudLab platform.
Then, they are stored as raw traces (i. e., the output of \texttt{EXPLAIN ANALYZE}).
Next, the authors preprocess the queries to remove very long-running queries (queries above 30\,s are excluded) to keep the benchmark practical and reproducible \parencite{zeroshot}.
A further preprocessing step is to parse the raw traces into structured plan representations.
These parsed plans (15000 per database) contain the operator nodes found in the raw traces, but organize them in a tree structure with node-wise attributes.
For every node, e.g., operator type, estimated/actual cardinalities, widths, and runtimes are stored.
The tree structure can be used to create pipelines and the node information will be used to train the model.
In the upcoming sections, we will describe these key implementation details.

Consequently, the parsed plans offer a good starting point for our pipeline and we can begin without further preprocessing.



\section{Implementation}\label{sec:methodology-implementation}

This section describes how a single parsed plan from the Zero-Shot benchmark is turned into the internal representation used by T3-style pipeline learning, and how we derive fixed-length feature vectors for each pipeline.
The procedure follows the pipeline idea of T3~\parencite{t3}, but replaces Umbra-specific tuple-size and stage-based featurization with quantities that PostgreSQL exposes in \texttt{EXPLAIN ANALYZE} traces.

\subsection{Mapping parsed PostgreSQL plans to the T3 plan format}

Each training example starts from one element of the \texttt{parsed\_plans} array in the benchmark JSON files.
Such an object is a rooted tree: every node carries \texttt{plan\_parameters} (operator name, estimated and actual cardinalities, tuple width, planner costs, optional filter predicates, timings, and further fields) and a list \texttt{children} of child subtrees.

We convert this tree recursively into the same nested operator structure that the original T3 implementation expects for Umbra plans~\parencite{t3}.
PostgreSQL operator names are mapped to abstract operators (\texttt{tablescan}, \texttt{join} with a physical variant such as hash or index nested-loop, \texttt{sort}, \texttt{groupby}, unary pass-through nodes, and temporary operators for hash build and materialize).
For joins, child order is normalized so that it matches T3's pipeline rules: for hash and merge joins the inner (build) subtree becomes the \emph{left} child and the outer (probe) subtree the \emph{right} child; nested-loop joins use the mapping required by the existing pipeline logic in the codebase.
Each node receives stable integer identifiers (\texttt{operatorId} and \texttt{analyzePlanId}) used later for pipeline membership.

\paragraph{Widths instead of per-column information systems.}
Umbra plans attach detailed information about produced columns (information units, IUs) and their sizes.
The parsed PostgreSQL plans do not provide an equivalent per-column breakdown; instead they report a single estimated tuple width per node (\texttt{est\_width}).
We therefore represent output size by one \texttt{producedIUs} entry per node whose \texttt{estimatedSize} is set from that width (with a small minimum), rather than summing many per-IU sizes as in the Umbra path.
Conceptually, this replaces fine-grained IU sizes with a single width feature per operator, which is what the benchmark offers.

Scan predicates in the parsed data appear as a nested \texttt{filter\_columns} tree (logical and/or/not and comparison-like leaves).
Where present, we translate this tree into the restriction format the T3 core understands (\texttt{expression}, \texttt{input}, optional \texttt{direction}), so that the same structural notion of a filter is available for any code path that still inspects restrictions.

To keep PostgreSQL-specific fields available after this normalization, every converted node additionally stores a copy of the original \texttt{plan\_parameters} under a dedicated \texttt{pg} payload.
Training and inference that use PostgreSQL-native features read only from this payload and from the pipeline layout, so we do not require a full database schema (table names in traces are not needed for that feature path; a minimal placeholder schema suffices for constructing \texttt{QueryPlan} objects in the implementation).

\subsection{Pipeline decomposition}

T3 groups operators into \emph{pipelines}: sets of operators that can produce tuples in a pipelined fashion, separated by \emph{pipeline breakers} such as sorting, grouping, or materialization~\parencite{t3}.
We assign each \texttt{analyzePlanId} to a pipeline id by walking the converted tree with the same high-level rules as in the T3 work: sort and aggregate nodes break pipelines; a materialize node is treated as a breaker; hash and merge joins place the build side in the preceding pipeline and keep the join together with the probe side in the next pipeline.

For every pipeline we record the list of operator ids, and we derive start and stop times from the per-node actual timings in \texttt{plan\_parameters} (converted to a common time unit) so that pipeline durations are available for training targets and bookkeeping.
The total query runtime label is taken from \texttt{plan\_runtime} when present.

\subsection{PostgreSQL-native feature vectors}

The original T3 feature mapper for Umbra builds a feature vector from \emph{execution phases} within each pipeline (e.g.\ scan, build, probe phases) and from cardinalities and tuple sizes attached to those phases~\parencite{t3}.
Relative to that design, our PostgreSQL adaptation differs in three main respects:
\begin{enumerate}
  \item \textbf{PostgreSQL-native operator features.} Parsed PostgreSQL plans expose one \texttt{EXPLAIN} node per iterator operator; we aggregate features by \emph{PostgreSQL operator name} (fixed layout per operator type) instead of by Umbra execution stage. This matches the iterator-based plan tree and avoids inventing synthetic build/probe phases for every node.
  \item \textbf{PostgreSQL planner cost estimates.} The pipeline-level block includes sums and extrema of PostgreSQL's \texttt{est\_cost} and \texttt{est\_startup\_cost} from \texttt{plan\_parameters}. These are optimizer-internal estimates of work and startup cost, not observed runtimes; they are absent from the Umbra-oriented feature path in the original T3 code.
  \item \textbf{Planned parallelism.} We sum \texttt{workers\_planned} over all operators in the pipeline so the model can see how much intra-node parallelism the PostgreSQL planner expected (zero on purely sequential plans).
\end{enumerate}
An excerpt of the resulting fixed-length vector (pipeline-level aggregates plus a slice of the per--operator-type block) is given in Figure~\ref{fig:pg-feature-vector-snippet}.

Concretely, we enumerate all operator types that occur in the parsed Zero-Shot plans (scans, bitmap scans, joins, hash and materialize build nodes, sort, aggregate variants, gather nodes, and others) and define a fixed feature layout.
For each pipeline the vector consists of two concatenated blocks.

In the Umbra implementation of T3, one pipeline row is obtained by summing feature vectors indexed by operator type \emph{and} execution phase (scan, build, probe, \ldots); pipeline scale is reflected inside those per-phase rows~\parencite{t3}.
PostgreSQL \texttt{EXPLAIN} nodes do not carry Umbra phases, so a single summed grid of that form is unavailable.
The \textbf{pipeline-level} block therefore collects these quantities as explicit pipeline-wide statistics: summed and extremal planner costs, total planned workers, mean tuple width, and aggregates over scan predicates and filter structure.
The \textbf{operator-type} block then supplies fixed entries per PostgreSQL operator name (counts, cardinalities, percentages), playing a role comparable to summing type-specific phase contributions in the original design.

\begin{enumerate}
  \item \textbf{Pipeline-level aggregates.} Over all operators in the pipeline we compute sums and extrema of actual cardinalities (falling back to estimates where actual values are missing), sums of PostgreSQL planner costs (\texttt{est\_cost}) and startup costs (\texttt{est\_startup\_cost}), the sum of planned parallel workers (\texttt{workers\_planned}), the average estimated tuple width, counts of operators by coarse class (scans, joins, sorts, aggregates, temporaries, pass-through), scan-specific sums and indicators (including whether scans have filters), counts of filter constructs (e.g.\ logical and/or, comparisons, \texttt{LIKE}, \texttt{IN}) derived from \texttt{filter\_columns}, the number of operators in the pipeline, and the root operator's output cardinality as a query-global signal.
  \item \textbf{Operator-type block.} For each PostgreSQL operator type in a fixed ordering, we accumulate a constant (occurrence count) and a small set of numeric descriptors (for example aggregated input and output cardinalities and widths, fractions relative to total scan cardinality in the pipeline, and expression-type fractions for scans), using definitions that match the join semantics of the converted tree (e.g.\ which child counts as probe stream versus build side for hash joins).
\end{enumerate}

Figure~\ref{fig:pg-feature-vector-snippet} illustrates how the two blocks are laid out end-to-end: pipeline-level entries summarize the whole pipeline (cardinalities, planner costs, parallelism, filters, \ldots), while the operator block repeats the same layout for every PostgreSQL operator type, filling in only the entries that correspond to operators present in that pipeline.

The layout is fixed so that every pipeline maps to one row of a feature matrix independent of plan size; operators that never appear contribute zeros in their entries.

In summary, we preserve T3's pipeline decomposition and per-pipeline prediction target~\parencite{t3}, while replacing Umbra-oriented tuple and stage features with a width-based size model and with PostgreSQL planner and cardinality signals derived from the parsed plan trees.

\section{Training}

This section describes how we fit the predictor to the Zero-Shot parsed plans.
We follow the tuple-centric training target proposed in T3~\parencite{t3}, but use the PostgreSQL-native feature matrix and the train--test splits implemented in our zeroshot training scripts.

\subsection{Per-pipeline feature rows and per-tuple targets}

Each training example contributes one row per pipeline: the feature vector is the corresponding row of the pipeline matrix from Section~\ref{sec:methodology-implementation} (one fixed-length vector per pipeline).
The label for that row is not the full pipeline duration directly, but the \emph{per-tuple pipeline time}, i.e.\ the observed pipeline duration divided by the pipeline's scan cardinality~\parencite{t3}.
Intuitively, the model learns how much time the pipeline spends per tuple entering from the scan side, which stabilizes learning across queries with very different result sizes.

Concretely, the scan cardinality of a pipeline is the sum of the cardinalities of all scan operators belonging to that pipeline (actual cardinalities from the parsed plan, with estimates as a fallback where needed).
At inference time, the model predicts a value for each pipeline row; that prediction is mapped back to a pipeline duration by multiplying with the same scan cardinality, and predicted query runtime is the sum of the predicted pipeline times – again as in T3~\parencite{t3}.

Training rows with an all-zero feature vector are dropped, because they carry no usable signal.

\subsection{Model and optimization}

We use gradient boosted decision trees as implemented in LightGBM~\parencite{lightgbm}, in line with the original T3 work~\parencite{t3}.
The booster is trained with a mean absolute percentage error (MAPE) objective on the training rows.
As in the reference implementation bundled with this project, each per-tuple target $t$ is clamped to a small positive floor, transformed as $-\log(t)$, and passed to LightGBM; predictions are mapped back with the exponential of the negative raw score before multiplication with scan cardinalities.

Following T3~\parencite{t3}, we cap the number of boosting rounds at \textbf{200} trees.
The motivation is the same as in the paper: a relatively shallow ensemble limits capacity and helps avoid overfitting to idiosyncrasies of the training workloads while remaining expressive enough for accurate runtime ranking and estimation.

\subsection{Data splits and reproducibility}

\paragraph{Generalization across benchmarks.}
For the main cross-benchmark evaluation, we use a \emph{leave-one-benchmark-out} split on the parsed-plan JSON files: all files whose path belongs to benchmark directories other than the held-out one form the training pool, and all files from the held-out benchmark form the test set.
This matches the intent of the Zero-Shot benchmark~\parencite{zeroshot} and of T3's cross-instance evaluation~\parencite{t3}, namely to measure whether the model transfers to an unseen database or workload folder.

\paragraph{Train and validation within the training pool.}
Inside the training pool, complete queries (entire plans) are partitioned at random into a training fraction of 80\,\% and a validation fraction of 20\,\%.
The random seed for this split defaults to \textbf{42} in the implementation.

\paragraph{Environment.}
Model training and evaluation for the results reported in this thesis were run on a personal laptop: Apple MacBook Pro (16-inch, 2021) with Apple M1 Pro (10 CPU cores, 16 GPU cores), 16\,GB unified memory, and 512\,GB SSD, under macOS 15.7.3.
